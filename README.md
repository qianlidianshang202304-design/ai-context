# AI Context 项目执行上下文

> 🧠 这是我的个人 AI 项目执行上下文仓库。它的目标不是“维护 Markdown”，而是让 AI 更快、更准地执行真实项目。

**第一原则：先干活，后沉淀。**

AI 读取这些文件，是为了理解我的项目、偏好、踩坑和验收标准，然后把当前任务做完。只有在项目执行完成、验收通过、产生了可复用经验之后，才把新信息回写到 Markdown。

---

## 🎯 仓库定位

这个仓库服务于三件事：

1. **执行项目**：帮助 AI 快速理解我要做什么、怎么做、做到什么程度算完成
2. **减少重复踩坑**：把真实项目中验证过的经验、字段、脚本、流程沉淀下来
3. **跨设备同步上下文**：不同电脑上的 AI 都能沿用同一套项目记忆和工作标准

这个仓库不服务于：

- 为了补全 md 而补全 md
- 为了整理文档而中断项目执行
- 在没有真实项目结果的情况下编造偏好、经验或最佳案例
- 把“写文档”当成最终交付

### 🧩 按需读取设计

为避免单文件过大、上下文过长，项目和案例采用「索引 + 子目录」结构：
- **索引文件**（`07-我的项目.md`、`10-项目经验.md`、`05-最佳案例.md`）：简短，始终可读
- **子文件**（`projects/`、`cases/`）：AI 按需深入，只读相关的
- **字典库**（`dictionaries/`）：稳定字段、标签、分类口径；涉及对应主题时必读
- **草稿区**（`drafts/`）：未获用户确认的新字典、新标签、新口径，只能暂存在这里
- **技能库**（`skills/`）：沉淀已跑通的执行流程；只有当前任务匹配时按需读取

## 🔄 正确工作流

```
用户给任务
  → 读取必要上下文
  → CEO 拆解：目标、边界、最小可交付
  → 驾驶舱执行：写代码 / 跑脚本 / 处理数据 / 生成交付物
  → 验收：用日志、截图、文件、测试结果证明完成
  → 仅当产生新经验时，回写 md
  → 追加一行对话日志
```

### 执行优先级

1. **项目交付物优先**：代码、脚本、报告、表格、截图、部署结果优先于文档整理
2. **可见证据优先**：用终端输出、运行结果、生成文件、浏览器验证证明任务完成
3. **经验回写靠后**：只有执行中发现了新的项目事实、偏好、踩坑、最佳案例，才更新对应 md
4. **不确定就标注**：无法确认的内容写 `[推测]`，不要把推测写成事实

### 什么时候更新 md

需要更新：

- 新增或确认了一个项目目录、技术栈、启动命令
- 发现了稳定的编码偏好或文件组织习惯
- 执行项目时踩坑并找到解决方案
- 有一段代码、脚本或流程被验证为最佳案例
- 用户明确要求同步上下文

不需要更新：

- 只是读了一遍规则文件
- 只是临时试错，没有可复用结论
- 当前任务还没执行完
- 没有经过文件、日志、运行结果验证

## 📝 日志机制

- **日常**：AI 在每次有价值的对话后追加一行到 `11-对话日志.md`
- **定期**：积累 10 条或 7 天后，AI 主动提炼到结构化文件
- **边界**：日志是项目执行后的沉淀，不是替代项目交付物
