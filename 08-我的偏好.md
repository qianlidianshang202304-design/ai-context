# ⚙️ 我的偏好

## 编码风格

### 通用规范
- **缩进**: Python 用 4 空格，JS/TS 用 2 空格
- **引号**: Python 用双引号，JS/TS 也偏双引号（JSX 属性用双引号）
- **分号**: Python 不加，JS/TS 一律加分号
- **换行符**: LF（macOS 默认）
- **文件末尾空行**: 需要（PEP 8 / Prettier 默认行为）

### 命名规范
- **变量**: Python snake_case，JS/TS camelCase
- **函数**: Python snake_case，JS/TS camelCase
- **类**: PascalCase（跨语言统一）
- **常量**: UPPER_SNAKE_CASE（跨语言统一）
- **文件**: Python snake_case，JS/TS kebab-case 或描述性名称

### 注释偏好
- [x] 需要写注释，解释为什么这样做
- [x] 分区用分隔注释 `# ----` / `// ======` 划分逻辑区块
- [x] 中文注释为主（前端和后端），英文注释出现在底层工具代码中
- [x] 「内容AI工厂」使用 emoji + 中文日志风格（如 `print("🚀 启动中...")`）
- [x] Python 文件顶部常写模块级 docstring，说明脚本目标、运行方式和边界
- [ ] JSDoc / 函数级 docstring 仅关键函数使用，非强制全覆盖

### 文件组织偏好
- Web 全栈项目倾向 `backend/` + `frontend/` 分离，后端用 `main.py`、`routers/`、`database.py`，前端用 `app/`、`components/`、`lib/`
- 数据/爬虫项目倾向单脚本或少量脚本解决问题，输出放 `outputs/`，缓存放 SQLite/JSON，最终交付 Excel
- 长任务脚本强调可恢复：`--refresh`、`--limit`、`--dry-run`、`--snapshot-output` 这类参数很常见
- 对外部系统字段名保持显式配置，例如飞书表字段、Excel 列名映射、API endpoint 都集中定义

## AI 交互偏好

### 代码生成
- **语言**: 中文解释 + 英文代码
- **风格**: 简洁直接，分点列表
- **注释**: 代码中的注释用中文（业务逻辑）或英文（底层工具）

### 回应风格
- **长度**: 简洁（不废话，直击要点）
- **结构**: 分点列表优先，复杂问题用表格
- **示例**: 需要代码示例

### 特别要求
- 不要假设我了解某个概念，如果不是基础概念请解释
- 给出多种方案时，请说明各自的优缺点
- 不要在我没有要求的情况下创建文件或执行命令
- 如果建议的命令有破坏性，请先警告我
- C 级任务先确认再动手

## 技术偏好

### 喜欢
- [x] 类型安全（TypeScript strict 模式，Python 类型注解）
- [x] FastAPI 异步风格（async/await）
- [x] 简洁的配置（约定优于配置，.env 管理敏感信息）
- [x] f-string（Python）和模板字符串（JS）
- [x] 用 pandas 做数据处理，openpyxl 做 Excel 读写
- [x] 飞书作为业务中台（多维表格存储、消息通知）
- [x] 大批量爬虫/数据任务先做缓存、断点续传、失败状态分类，再追求速度
- [x] 浏览器自动化优先复用登录态，避免每次重新登录或触发风控

### 不喜欢
- [x] 过度设计（"万一以后要用"的抽象）
- [x] 不必要的抽象层（能用一个函数的不用一个类）
- [x] 魔法值（magic numbers/strings）
- [x] any 泛滥（TypeScript）或无类型注解的复杂 Python 函数；已有代码中少量 `any` 主要用于动态 API 参数和外部返回值

### 其他观察
- 前端有 `eslint.config.mjs`，未看到独立 Prettier 配置；Python 项目未见统一 Ruff/Black 配置，主要靠个人习惯保持一致
- Python 和 JS/TS 的缩进宽度不同（4 vs 2），但各自领域内严格遵守
