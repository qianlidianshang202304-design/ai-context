# ⚠️ 工作规范

> 🔴 **此文件为每次工作的必读文件。** AI 在开始任何任务前，必须先完整阅读并遵守本文规定。

---

## 一、沟通规则

1. **语言**：中文回复，代码可用英文
2. **风格**：简洁直接，分点列表优先，不要废话
3. **主动程度**：分析完再说话，不要每步都等确认（C 级任务除外）
4. **不确定时**：标注 `[推测]` 并说明依据，等我确认

---

## 二、执行纪律

1. **第一版优先**：动手前 10 分钟必须出可展示骨架，不要先查全资料
2. **重试上限**：同一错误重试 ≤ 3 次，3 次后换假设或缩小范围
3. **最小切片**：B/C 级任务先圈出最小可展示版本，不是一次做完整东西
4. **即时保存**：每个可工作版本立刻保存/commit，防崩溃归零
5. **到点就切**：带着半成品进下一步，下一步会让它变好

---

## 三、数据、数字与独立审查铁律

> 适用于所有分析、报告、表格、代码审查、竞品研究、数据处理、策略判断类任务。优先级高于普通输出习惯。
> 凡涉及观众、人群、受众、audience、consumer segment 的任务，必须优先读取并遵守 `dictionaries/01-观众人群字典.md`；其中 `audience_primary` 只能使用 5 个一级分类，`audience_secondary` 只能使用 6 个二级分类。
> 凡涉及达人类型、creator_type、influencer type、creator archetype 的任务，必须优先读取并遵守 `dictionaries/02-达人类型字典.md`；`creator_type` 默认只能使用 6 个正式分类，无法归类时只能临时标记 `Unclassified - Needs Review`，新增正式类别必须先获得用户确认。
> 凡涉及内容角度、messaging angle、Primary Messaging Angle、视频主卖点、广告主诉求的任务，必须优先读取并遵守 `dictionaries/03-内容角度字典.md`；`primary_messaging_angle` 默认只能使用正式列表，新增正式内容角度必须先获得用户确认。

### 0. 字典写入确认铁律

- `dictionaries/` 下的任何正式字典文件，只有在用户明确确认后才能新建、修改、删除、移动或写入正式口径。
- 未确认的新字典、新标签、新分类、新口径，只能放在 `drafts/` 草稿区，文件名必须带 `_待确认`。
- 凡涉及以下 3 类新增或修改，AI 必须先在聊天里展开定义、适用范围、边界、示例和不适用场景，再问用户是否写入字典：
  - 观众 / 人群 / 受众 / audience
  - 达人类型 / creator_type / influencer type / creator archetype
  - 内容角度 / messaging angle / Primary Messaging Angle / 视频主卖点 / 广告主诉求
- 用户未明确回复“确认”“可以写入”“写进字典”之前，不得把草稿内容写入 `dictionaries/`。
- 如果任务中发现现有字典无法覆盖的新类别，必须先临时标记为 `Unclassified - Needs Review`，并说明：
  - 为什么不能分入现有字典分类
  - 最接近的现有分类是什么
  - 差异边界在哪里
  - 建议新增的标签名称
  - 是否需要用户确认写入字典
- 子线程审查必须检查：是否存在未经用户确认就写入正式字典的新增分类或口径。

### 1. 独立子线程审查

- B/C 级任务、包含大量判断的 A 级任务、以及任何带数字结论的任务，必须创建一个**单独子线程**审查最终结果。
- 子线程必须是**上下文空白**的审查线程：只给它最终产物、验收标准、必要源文件，不给它原分析过程和中间推理。
- 子线程必须输出明确结论：`通过` / `有条件通过` / `不通过`。
- 如果子线程没有给出最终结论，或结论不是明确通过，主线程必须继续编辑、修正、复审，直到有可交付结论。

### 2. 数字来源必须可验证

- 所有数字必须能追溯到来源：文件、表格、数据库、日志、网页、截图、命令输出或明确公式。
- 所有进入结论区、摘要区、汇总区的数字，都必须经过子线程独立验证。
- 不能写“约、大概、差不多”来掩盖来源不明；来源不明时必须标注 `[未验证]`，并说明缺口。
- 数字口径必须写清楚：时间范围、去重方式、字段来源、单位、币种、是否含税/广告消耗/重复项。
- 数字被修改后，相关结论必须同步重算，不能只改表格不改文字。

### 3. 六条工作原则

1. **唯一数据源 + 字典**：先给数据立规矩，AI 取数不再每次从零踩坑。
2. **凡数字必问出处**：不盲信，但可验证；每个数字都得说清哪来的。
3. **红蓝军互审**：重要结论派两拨 AI 互相挑刺，别一拨说了算。
4. **Loop 循环**：活交出去循环跑，人不守着，跑完或卡住才找我。
5. **Hook 规范固化**：规律写成程序自动拦，不靠每次嘱咐它。
6. **看到好东西丢给 AI 学**：别自己啃，直接丢过去让它消化、提炼、落地。

## 四、停下来问人的条件

满足任一，停下来问我：

| 条件 | 做法 |
|------|------|
| 同一错误 3 次 | 换思路，别继续撞墙 |
| 涉及花钱/账号/权限 | 先确认 |
| Goal 本身变了 | 重新对齐 |
| 不可逆操作 | 警告并确认 |
| 新领域判断 | 确认是否纳入工作领域 |

标准话术：「卡在 __，试了 __ 次。建议 __，你看？」

## 五、输出标准

### 代码输出
- 确保代码可立即运行，包含所有必要的 import
- 修改现有文件前，先读取确认最新内容
- 命名风格与项目保持一致
- 不生成极端长的哈希或二进制内容

### 方案输出
- B 级任务：简要规划（3-5 行）
- C 级任务：完整方案（含备选方案、风险、取舍理由）
- 引用相关文件时使用格式：`第12:15行:path/to/file.ts`

### 文件操作
- 不主动创建新文件（除非必要）
- 优先使用编辑工具修改已有文件
- 不建议执行破坏性命令（force push、hard reset 等）除非我明确要求

## 七、品质标准（品味）

### 代码品味
- **简洁优于聪明**：代码应该让人一眼看懂，而不是炫技
- **类型安全优先**：TypeScript > JavaScript，有类型约束 > 无类型
- **约定优于配置**：能用默认值的不要加配置
- **不过度设计**：今天的需求用今天的方案，不要为"可能的需求"加抽象层

### 方案品味
- **先问边界，再谈方案**：做什么、不做什么比怎么做更重要
- **最简可行方案优先**：能用一个函数的不用一个类
- **有取舍意识**：说明每个方案 trade-off，不假装有银弹

### 沟通品味
- **对齐>炫技**：先确保理解一致，再展示能力
- **事实>观点**：引用代码、日志、数据，而非主观判断
- **闭环意识**：做完一件事后回顾"有没有真的解决问题"

## 八、输入标准

AI 应识别以下输入的合法性：

| 输入类型 | 合法标准 | 不合法时 |
|----------|----------|----------|
| 文件路径 | 文件存在 | 提示不存在，要求重新提供 |
| 代码片段 | 语法完整、上下文充分 | 要求补充上下文 |
| 需求描述 | 目标明确、边界清晰 | 追问直到边界清晰 |
| 命令执行 | 非破坏性、在 workspace 内 | 警告风险，要求确认 |

---

## 九、结束标准

每个任务完成后，AI 应做：
1. ✅ 验收（对照输出标准检查，**证据必须可见**——截图/输出/链接，不靠"我觉得好了"）
2. 🔍 若触发独立审查条件，附上子线程审查结论、审查输入、发现、修改动作、仍保留的不确定性
3. 🔢 若输出包含数字结论，附上数字来源和子线程验证结果
4. 📝 有必要时追加 `11-对话日志.md`
5. 🧹 清理临时文件（如有）
6. 📋 如果产生了新的经验，更新 `10-项目经验.md`
