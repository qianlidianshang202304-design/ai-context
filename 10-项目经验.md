## 🔥 通用踩坑总结

### 坑 1：Excel 大文件性能问题
- **场景**: 10万+ 行数据用 openpyxl 读写
- **解决**: 大文件用 CSV 中间格式，最终报告用 xlsxwriter
- **教训**: 选对工具比优化代码重要

### 坑 2：AI API 调用成本控制
- **场景**: Claude API 批量处理数千条数据
- **解决**: 加本地缓存层（JSON/sqlite），已处理过的跳过
- **教训**: 能用规则解决的不用 AI

### 坑 3：飞书 API 限流
- **场景**: 批量同步到飞书多维表格
- **解决**: 延时 + 分批 + 失败重试
- **教训**: 第三方 API 必须考虑限流策略

### 坑 4：TikTok 页面抓取失败不能都当终态
- **场景**: 大规模抓 TikTok Bio 时遇到 `no_universal_data`、403、429、bot challenge
- **解决**: 把失败状态分为可重试和不可重试，可重试状态不写入最终缓存
- **教训**: 爬虫缓存要缓存“可信结果”，不要把临时失败固化

### 坑 5：飞书同步必须字段级验证
- **场景**: Amazon Creator 消息扫描写入 `扫描回复情况`、`BD评估 BD tips`、`发信日期 sending date`
- **解决**: 写入前校验真实表 ID 和字段名，结果汇报 parsed / matched / unmatched / write_failure
- **教训**: 外部表格同步不能只报“成功”，要验证关键字段是否真的被写入

### 坑 6：社媒页面自动化依赖登录态和低频策略
- **场景**: IG / WotoHub / ZiNiao 页面抓取容易遇到弹窗、空白页、懒加载和风控
- **解决**: Playwright 复用 `user-data-dir`，滚动触发懒加载，夜间暂停和随机间隔
- **教训**: 浏览器自动化的稳定性优先级高于速度

## 🧠 我的设计原则

1. **最简可用先行**：不为了"以后可能"的需求加复杂度
2. **AI 是工具不是银弹**：把 AI 预算花在刀刃上
3. **数据管道要幂等**：脚本必须能重复运行不产生副作用
4. **内部工具不追求完美 UI**：功能可用 > 界面精美
5. **字段和来源要可追溯**：输出表要保留来源、状态、异常原因，方便复核
6. **代理证据要标清**：官方字段拿不到时，可以用 proxy 标签，但不能伪装成官方结论
