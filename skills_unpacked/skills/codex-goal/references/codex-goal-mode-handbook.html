Codex Goal Mode
白话操作手册

Goal 模式 = 给 Codex 下一张「工单」。
写清楚要做到什么、怎样算做完，然后让它自己一步步推进，直到验收通过。
和普通对话的区别 普通对话：你问一句，它答一句。
Goal 模式：你定好终点和验收标准，它可以连续做多步（改代码、跑测试、再改、再测），中间不用你每步都催「继续」。
最重要的一句 Goal 不是「让它一直自动跑」。它是一份有边界的完成契约：目标清楚、范围清楚、什么时候必须停下来问你，都要事先写好。

什么时候该开 Goal？

适合开

  • 任务要好多步才能做完
  • 需要反复试、反复跑测试
  • 你会一直说「继续修」「再试一种」
  • 需求已经想清楚，只是执行很长
    （比如重构、修 bug、按文档改完）

不适合开

  • 问一句话、改一行代码
  • 让它解释概念、做头脑风暴
  • 你自己也还没想好要什么
  • 每一步都要你拍审美或方向
判断口诀 如果你会反复说「继续，直到真的完成」→ 开 Goal。
如果只是问完就走 → 普通对话就够了。

怎么用：七步法

1

先判断值不值得开

长任务才开 Goal，短问题用普通聊天。

开错了 Goal 会浪费额度和时间。

2

开工前想清楚五件事

目标、背景、不能动什么、怎样算完成、什么情况必须停下来问你。

需求模糊时，先用 /plan 聊清楚再开 Goal。

3

写 Goal 正文

按「要做什么 + 怎样验收 + 有什么限制 + 何时暂停」来写。

别说「优化一下」「重构一下」这种没法判断是否完成的话。

4

给它能快速验证的办法

写清楚：跑哪些测试、看什么数字、截什么图、读哪份检查清单。

有客观标准，Codex 才能自己判断「做对了没有」。

5

准备一份外部记录文件

长任务让它把计划、尝试、失败原因写进一个文档（如 PLAN.md、进度笔记）。

对话太长会被压缩，外部文件帮它「记住」之前做了什么。

6

运行中盯着别跑偏

中途可以用普通消息纠偏。如果它反复问同一件事、钻牛角尖、或额度烧太快 → 输入 /goal pause 暂停，理清后再继续。

长跑任务容易「修小问题修上瘾」，忘了最初目标。

7

结束前你自己验收

不要只看它说「完成了」。你要看：代码改动、测试结果、还有没有漏项、有没有风险。

它停了或额度用完了，不等于任务真的做完了。

直接复制的模板

把方括号里的内容换成你的任务，整段粘贴给 Codex 即可。

推荐模板（中文注释版） 
/goal 在【项目/文件夹范围】内完成【具体结果，要能判断是否做完】。

验收标准：必须用【测试通过 / 截图 / 数据对比 / 人工检查清单】证明完成。

限制：不要动【哪些文件或模块】；保持【代码风格 / 兼容性 / 安全要求】。

过程中：把每次尝试、失败原因、剩余风险记录到【PLAN.md 或进度文档】。

暂停条件：遇到【需要我拍板 / 要花钱 / 事实查不清 / 同一问题卡住 3 次】时，
停止循环，列出需要我决定的事项。

完成前自审：检查代码改动、测试结果、未完成项、风险、下一步建议。

三个真实场景示例

查研究报告里的说法是否靠谱

/goal 核查这篇 AI 产品报告里每个关键说法是否有依据。
列出：说法清单、证据链接、能确认 / 只能部分支持 / 查不到 / 不确定 四类结果。
查不到的事实不要硬写结论，停下来问我。

把会议速记写成文章

/goal 根据附件速记写活动回顾初稿。
先整理素材对应关系，再写正文；保留嘉宾原话，不编造观点。
写完检查：事实、标题、开头、金句是否准确。

把成功流程写成可复用文档

/goal 把这次跑通的工作流整理成标准操作文档。
必须写清：什么时候用、需要什么输入、步骤、怎样验收、常见错误、维护边界。
如果要改系统配置文件，先暂停问我。

常用命令

命令干什么
/goal开始或设置一个 Goal
/goal pause暂停，先缓一缓
/goal resume从暂停处继续
/goal clear取消当前 Goal
/plan需求还不清楚时，先聊计划（再开 Goal）

术语白话翻译

原文档里有些英文词，这里用一句话解释：

blocker
卡住的地方，必须有人决定才能继续
evidence
证据，能证明「做完了」的东西（测试、数据、截图）
spec
需求文档，已经把要做什么写清楚了
tunnel vision
钻牛角尖，沉迷修小细节，忘了大目标
steering
中途纠偏，运行中用普通消息调整方向
budget
额度/用量上限，用完了会停，但不代表任务完成
claim audit
事实核查，逐条检查文章里的说法有没有依据

常见坑（社区真实反馈）

  • 同一问题反复问你：在 Goal 里写「同一卡点 3 次就暂停，列清单给我」
  • 跑太久额度烧光：长任务拆小 Goal，或设检查点
  • 对话压缩后变笨：关键进度写进外部文档，别只靠聊天记忆
  • 修小问题修上瘾：每隔几轮提醒它「回到最初目标」
  • 它说完成了但其实没有：你必须自己看测试和代码改动

写 Goal 前快速检查

1. 目标做完后具体得到什么？能不能判断是否完成？
2. 范围改哪里？哪里绝对不能动？
3. 限制风格、兼容性、安全有什么要求？
4. 验收跑什么测试？看什么数字？谁来最终确认？
5. 暂停什么情况必须停下来问你？
