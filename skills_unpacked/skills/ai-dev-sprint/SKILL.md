# AI Dev Sprint · 时间盒开发

## 产出物

在时间盒内（个人 30min / 组队 60min）做出一个**可展示的运行成果**：网页 / App / 工具 / 脚本 / 自动化 / HTML demo / 数据报告。

## 不负责

- 不写目标契约（先用 `codex-goal` 拿到 Goal Brief）
- 不做展示包装（→ `demo-deck`）
- 不做组队（→ `team-match`）

## 五步链（Goal → 学习 → 开发 → 测试 → 收口）

| 步  | 动作                | 出什么                                                    | 通过标准                         |
| --- | ------------------- | --------------------------------------------------------- | -------------------------------- |
| 1   | 接 Goal             | 确认陈述句终态 + 验收                                     | 知道「怎样算完成」               |
| 2   | GitHub 学习（必做） | 找 1 个相关参考仓库，逐模块问「脱离它还有用吗」，只抽能力 | 不 clone、不照抄，只拿可复用模式 |
| 3   | 最小骨架            | 先跑通能展示的核心                                        | 浏览器 / 终端能看到东西          |
| 4   | 迭代 + 测试         | 每步可验证（跑命令 / 打开页面 / 看输出）                  | 验收命令通过或截图为证           |
| 5   | 收口                | diff + 未完成项 + 风险 + 下一步                           | 留 Demo 接口                     |

## 第 2 步：GitHub 学习模式（必做，不安装不 clone）

开做之前先学一个相关的开源项目，这是工坊的硬动作，不是可选项。哪怕只花 5 分钟，也要找一个跟你的 Goal 沾边的 GitHub 仓库，从它身上抽一个能用的做法，再开始写。理由：站在别人趟过的路上做，比从零硬憋快得多，这正是「干中学」。

一句话目标：不是评价这个仓库厉不厉害，而是抽出**脱离原仓库后仍然独立有用的能力**，注入你自己今天的成果。

### 硬规则

- 不安装、不 clone、不本地跑——只读它的 README / 关键目录 / 核心代码片段
- 不做泛化竞品分析，只看能力机制
- 逐模块判断复用价值：不能脱离原仓库独立成立的，直接跳过
- 只抽「脱离仓库后仍有用的能力」，不整包搬运

### 四步法（贴一个仓库，让 AI 走一遍）

1. 快速扫描：这个项目本质在做什么？核心机制是什么？哪些目录最值得看？值不值得继续拆？
2. 逐模块拆解：每个核心模块只问三件事——你的成果里已有对应物吗？脱离这个项目后还独立有用吗？该接到你成果的哪个位置？
3. 只提炼增量：把可复用的机制抽出来，明确接入点，直接用进你的最小骨架。
4. 不采用也写清楚：列出不抄的模块和原因。判断力的一半来自「不要什么」。

### 禁止

- 「这个项目很有参考价值」这类空话；把包装当机制
- 因为 star 多就高看、star 少就忽略
- 顺手做完整 code review；讨论市场地位却不落到能力抽离

完整方法与执行链见包内本地副本（在别人电脑上也能用，不依赖余一工作区）：

## 时间盒纪律（AI First）

- 前 10 分钟必须让 AI 出第一版可展示骨架，不先查全资料
- 同一错误重试 ≤ 3 次，换假设或缩小 Goal
- 优先「能 Demo 的版本」，polish 留到组队 Build
- 每个可工作版本即时保存 / commit，防崩溃归零

## 测试与验证（防假完成）

| 类型        | 怎么验                     |
| ----------- | -------------------------- |
| 网页 / HTML | 浏览器打开，关键交互点一遍 |
| 脚本 / 工具 | 跑一次，看输出 / 退出码    |
| 自动化      | 触发一次，确认副作用正确   |
| 数据 / 报告 | 抽样核对，不编造数字       |

证据必须可见（截图 / 输出 / 链接），不靠「我觉得好了」。

## 停下来问人的条件

- 需要花钱 / 改账号 / 发布到外部
- 连续 3 次同一 blocker
- Goal 的终态本身变了

## 组队 Build：基于 Demo 1 反馈再优化（段 4 复用）

个人 Build（段 2）出第一版，组队 Build（段 4）是「拿第一版换来的反馈，做出更好的下一版」。这一段不重新开题，先把 Demo 1 的成果和反馈接住：

| 步  | 动作        | 通过标准                                                   |
| --- | ----------- | ---------------------------------------------------------- |
| 1   | 收反馈      | 把 Demo 1 听到的提问 / 质疑 / 合作信号列成清单             |
| 2   | 定优化方向  | 选深化 / 合并 / 开新题之一，写清「这一版要比上一版多什么」 |
| 3   | 改 Goal     | 把优化方向写回 Goal 的终态和验收，别凭感觉改               |
| 4   | 迭代 + 测试 | 同样小步快跑，每个版本可验证、即时保存                     |
| 5   | 收口        | 团队版 diff + AI 杠杆点 + 24h 动作，留 Demo 2 接口         |

铁律不变：先出能展示的版本，再 polish；到点停，带着团队版进 Demo 2。

## 验收

- [ ] 时间盒内有一个可展示运行成果
- [ ] 用了 Goal，知道怎样算完成
- [ ] 做了 GitHub 学习：找了 1 个参考仓库，抽了能力、写清不采用什么（只抽能力未照搬）
- [ ] 有可见的测试 / 验证证据
- [ ] 留了 Demo 接口（做了什么 / 感受 / 想组队方向）
