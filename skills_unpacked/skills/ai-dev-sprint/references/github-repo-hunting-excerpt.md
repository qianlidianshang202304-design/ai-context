## GitHub 仓库狩猎（专项流程）

余一给 GitHub 仓库链接、README、项目主页，或说"看看这个项目""分析一下这个仓库""有什么值得学的""拆解一下"时，进入此子流程。

**一句话目标**：不是评价仓库厉不厉害，而是抽离出**脱离原仓库后仍然独立有用的能力**，注入余一系统。

### 硬规则

- 不安装，不 clone，不本地跑
- 不做泛化竞品分析，只看能力机制
- 逐模块判断复用价值：不能脱离原仓库独立成立的，直接跳过
- 每个增量必须映射到自己的系统：rule / skill / 执行纪律 / lessons-learned 之一

### 流程

1. **快速扫描** -- 回答四个问题：解决什么问题？核心机制是什么？哪些目录最值得看（skills/rules/hooks/prompts/commands/evals）？值不值得继续拆？
2. **逐模块拆解** -- 每个核心模块只问三件事：我们已有对应物吗？脱离这个项目后还独立有用吗？应该注入到我们系统的哪个位置？
3. **生成覆盖/增量评估表** -- `| 模块 | 我们的对应物 | 覆盖程度 | 增量价值 |`
4. **只提炼增量** -- 提炼成可执行机制，明确注入位置，执行注入或给出精确改动点
5. **不采用也写清楚** -- 列出不采用的模块和原因。判断力的一半来自"不要什么"

### 交付格式

1. 一句话说完：这个项目本质在做什么
2. 已覆盖 vs 增量：评估表
3. 注入了什么：具体文件变化或注入点
4. 不要什么：不采用的部分 + 原因

### 禁止

- "这个项目很有参考价值"这类空话
- 把包装当机制
- 因为 star 多就高看，因为 star 少就忽略
- 顺手做完整 code review
- 讨论市场地位却不落到能力抽离

### 与其他 Skill 的关系

- `github` skill：执行入口，负责 gh CLI、仓库元信息查询和 GitHub 平台操作
- `skill-onboarding`：当增量值得形成独立 skill 时进入
