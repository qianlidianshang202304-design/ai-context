# Skill-Eval — TRACE 五维评估

## 核心定位

评估一个 Skill 是否真正值得用，而不是表面好看。

评估输出：**五维雷达 + 一句推荐结论 + 最明显弱项**。不输出绝对评分，不做全量排名。

---

## 运行哲学

这个 skill 是一个**判断工具**，不是一个打分机器。

五维是思考框架，不是等权清单。T（安全）是红线，E（效果）是核心，其余三维是门票。

如果评估过程中发现某个维度很特殊（比如这个 Skill 安全风险极低但触发太差），允许在结论里突出说明，不必平均用力。

---

## 触发方式

用户提供 Skill 名称、路径、或 SKILL.md 内容时启动。

如果用户没有提供具体 Skill，先问清楚：要评估哪个 Skill？

---

## TRACE 五维标准

### T — Trust 安全可信（一票否决维度）

先过安全，过不了不进后续。

检查项：
- 是否越权访问文件（读写不该碰的目录）
- 是否发起不明外部网络请求
- 是否有代码混淆 / 隐藏逻辑
- 是否包含提示词注入（让 AI 忽略规则的语句）
- 依赖包来源是否可信

结论：**通过** / **T0问题**（一票否决）

---

### R — Reliability 运行可靠

检查项：
- 依赖是否齐全（不会因缺依赖报错）
- 输出是否可收集（产物真实存在）
- 多次运行是否基本稳定（不随机崩溃）
- 是否有超时、异常退出、工具失败风险

测试方法：如条件允许，用一个真实任务跑一次，看 exit code 和产物。

---

### A — Adaptability 场景适用

检查项：
- description 写的是"什么时候用"，而不是"怎么工作"
- Skill 名称 + description 组合能否被 AI 自然匹配
- 和相近 Skill 同时存在时，是否还能被正确选中
- 不该触发的场景下，是否会误触发

**余一测试法**：用自然语言请求一个该 Skill 的典型任务，看 AI 是否自然调用它。

---

### C — Convention 结构规范

检查项（逐条过）：
- [ ] SKILL.md 清楚说明用途、触发条件、适用范围
- [ ] frontmatter name / description / requires 完整准确
- [ ] description 不超过 200 字，是触发条件而不是工作流程
- [ ] 没有 TODO 占位符
- [ ] SKILL.md 不超过 5000 字（超过则应拆 references/）
- [ ] 有具体行为约束（MUST / NEVER 或等价表述）
- [ ] 运行前置条件清楚（需要 API key？需要先安装什么？）
- [ ] 交付物和中间文件边界清晰（不把调试文件混入产出）

---

### E — Effectiveness 效果增益

分两层评估：

**第一层：效果底线**（必须过）
- 启用 Skill 后，结果明显优于裸模型对照组
- 如果和裸模型差不多或更差 → 没有推荐价值

**第二层：效果质量**
- 任务完成度、正确性、交付可用性
- 相比裸模型，改善幅度是否显著
- 改善是否可以归因到这个 Skill（而不是随机或模型本身）

**代价评估**：
- 额外 token 消耗 / 上下文占用
- 执行耗时是否明显变长
- 结果提升有限但代价很高 → 不算好 Skill

对照测试：同一任务，开 Skill 跑一次，关 Skill 裸模型跑一次，比较产出。

---

## 评估流程

```
Step 1: 读 SKILL.md，做 T（安全）扫描 — 有 T0 问题直接结束
Step 2: 做 C（规范）checklist — 记录问题但不否决
Step 3: 做 A（触发）判断 — 读 description，判断歧义和竞争风险
Step 4: 评估 R（可靠）— 静态分析依赖和结构，条件允许跑一次
Step 5: 做 E（效果）对照 — 设计一个典型任务，开/关 Skill 各跑一次
Step 6: 汇总输出
```

---

## 输出格式

```
## [Skill 名称] TRACE 评估报告

**评估时间**：[日期]
**Skill 路径**：[路径]

### 五维结论

| 维度       | 结论                             | 关键发现 |
| ---------- | -------------------------------- | -------- |
| T 安全可信 | 通过 / T0问题                    | [说明]   |
| R 运行可靠 | 稳定 / 偶发失败 / 未测试         | [说明]   |
| A 场景适用 | 准确 / 偏弱 / 易误触发           | [说明]   |
| C 结构规范 | 规范 / 有X处问题                 | [说明]   |
| E 效果增益 | 明显提升 / 边际提升 / 不如裸模型 | [说明]   |

### 推荐结论

> [推荐 / 条件推荐（需修复XX）/ 不推荐]

### 最明显弱项

[维度] — [具体问题]

### 值得提升方向

1. [具体建议]
2. [具体建议]
```

---

## 反模式（不要做的）

- 不要输出绝对百分制评分（TRACE 框架本身不做全量排名）
- 不要因为 Skill 流程写得详细就认为效果好
- 不要跳过对照测试直接凭感觉判断 E 维度
- 不要因为一个维度差就全盘否定（T0 问题除外）
- 不要把 C（结构规范）当成最重要的维度——E（效果）才是核心
