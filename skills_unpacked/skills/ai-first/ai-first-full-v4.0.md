# ai-first-redesign v4.0

> 注意：这是余一内部的**全量深挖版存档**，含 `.yuyi/`、`02AI 余一/` 等余一工作区专属路径，**工坊现场不用、别人电脑上不保证可跑**。工坊参与者走同目录的轻量版 `SKILL.md`。本文件仅供会后想做完整全人 AI-First 审计时参考。

> 帮一个人把自己摊开。然后用真方案 + 真红灯告诉他下周该做什么。
>
> v4.0 核心变化：不让用户描述工作流。AI 基于上下文推理出步骤级工作流框架，用户只确认/修正。先拆一条校准标尺，再以此为标准跑全部。

## 核心定位（不要走偏）

这个 skill **不是**"拆一条工作流的工具"。是"摊开一个具体的人 + 给真方案"的工具。

- 一条工作流只是入口或样本
- 真正的产物是这个人的"**全人 AI First 重做报告**"
- 企业模式是 N 个全人报告的组合，不允许凭空画职能图
- **每条活儿的产物是步骤级文件系列**：一个步骤一个主文件，禁止按原子继续拆子文件

如果你（AI）发现自己只在拆一条流程、或把所有内容堆在对话里，立刻停下回到这一节。

## 两级粒度定义（贯穿全流程）

|              | 步骤级                           | 原子级                                        |
| ------------ | -------------------------------- | --------------------------------------------- |
| **定义**     | 一个可识别的工作阶段/动作块      | 不可再拆的最小执行单元（四门全过）            |
| **粒度**     | "收集竞品信息""写初稿""内审修改" | "打开XX数据库→输入关键词→筛选近30天→逐条摘录" |
| **数量**     | 一条工作流大概 4-8 步            | 一个步骤可能拆出 3-10 个原子                  |
| **判断标准** | 用户能说"对，我确实有这个阶段"   | 能映射到一次工具调用/一次人判断（四门全过）   |
| **用途**     | Step 2 让用户确认框架            | 阶段 2 让 AI 评估每个动作能否被替代/增强      |
| **谁产出**   | AI 推理 + 用户确认/修正          | AI 拆解（基于原子拆解标准四门）               |

**从步骤到原子的过渡**：Step 2 框架确认后，阶段 2 才把步骤拆到原子。用户不需要参与原子拆解过程——他已经在步骤级确认了框架对不对。

## 信息补足机制（v1.2 新增）

> 信息不足时**不跳过**，也**不只问用户**。用多种方式让信息充足起来。

详见 [references/信息补足机制.md](references/信息补足机制.md)。四条路径概要：

1. **推理填充 + 用户确认**：从上下文推理生成完整内容，标注来源，让用户改不对的
2. **结构化推理（同类外推）**：用已知原子的模式外推未知原子，标注推理依据
3. **模型知识 + 可验证标记**：用模型知识给初稿，禁写 URL，必附验证步骤
4. **分阶段交付**：标明缺口和影响，把"补全"变成 MVP 动作

**决策原则**：能推理就推理（标注 + 确认），不能推理才标缺口。绝对禁止不标注就跳过。

## v1.0 三条硬约束（不可违反）

### 硬约束 1：真对话约束

访谈必须真问真等。**禁止**：

- ❌ 一次抛全部问题
- ❌ 自填字段（即使在测试模式扮演用户也不行）
- ❌ 跳段
- ❌ 用户没回答就生成"访谈记录"

详见 [references/A路径访谈脚本.md](references/A路径访谈脚本.md) 的"真对话约束"小节。

### 硬约束 2：subagent 并行搜方案

市面方案搜索强制并行。**禁止**：

- ❌ 串行调用 web-access（10 原子串行 = skill 跑不动）
- ❌ 凭记忆吐方案
- ❌ 只给推荐不给搜索过程和淘汰名单
- ❌ **搜索失败/超时后降级为凭记忆补位并标注"未验证"**——这不是降级，是违规

**搜索失败处理硬门**：
1. 单个 subagent 搜索失败 → 重试 1 次（换关键词/换搜索引擎）
2. 重试仍失败 → 该原子方案区写"搜索未成功，待补验证"，**不写任何具体方案名/URL**
3. 所有 subagent 全部失败 → 整个阶段 3 标记为"未完成"，不允许进入阶段 4-8，要求用户决定是否刷新快照或手动提供信息

**绝对禁止**：搜索失败后用模型已有知识补位写出"推荐方案名 + URL + 能力描述"——这等于凭记忆吐方案，无论是否加 ⚠️ 标记都是违规。

**环境降级链（v1.2 新增）**：

当运行环境不满足"并行 subagent 派发"条件时，按以下优先级逐级降级：

| Level         | 条件                           | 做法                                                                              |
| ------------- | ------------------------------ | --------------------------------------------------------------------------------- |
| L1 理想态     | 可并行派发 subagent            | 按规约并行搜索，等返回，验收                                                      |
| L2 部分能力   | subagent 不可用但有 web-access | 主 agent 用 web-access 串行搜索 **Tier 1 原子**（痛点区 ≤ 8 个），非痛点原子走 L3 |
| L3 无搜索能力 | 完全无法联网搜索               | 见下方"L3 合规替代"                                                               |

**L3 合规替代（不是"加 ⚠️ 标注"）**：
1. 市面方案区写"**模型推理生成，未经搜索验证**"
2. **禁止写 URL**（URL 是搜索结果，不是推理产物）
3. 可以写方案名 + 能力描述，但每个方案**必须附验证步骤**：具体搜索词 + 验证方法 + 判断标准
4. `04_AI-First重做.md` 红灯清单增加一条："方案搜索未执行"
5. MVP 第一步改为"补跑方案验证"——给出具体的搜索关键词组合
6. `00_概要.md` 显著标注："本次评估在 L3 模式下完成，市面方案为推理生成，需补验证"

L3 与违规的本质区别：L3 不伪装为搜索结果（无 URL），把验证变成用户的第一个可执行动作（而非脚注），且在概要和红灯中显著标记。

详见 [references/搜索方案subagent调用规约.md](references/搜索方案subagent调用规约.md)。

### 硬约束 3：步骤级文件输出

**禁止**：

- ❌ 把所有内容堆在对话里
- ❌ 一个 markdown 文件全装下
- ❌ 每个原子单独建评估文件或搜索结果文件
- ❌ AI First 拆成演进版 / 重构版 / 红灯 / MVP 多个文件

**强制**：每条活儿产生 5 个步骤文件，落到 `02AI 余一/工作流审计/[活儿名]_[YYYY-MM-DD]/`。

| 文件                   | 内容                                        |
| ---------------------- | ------------------------------------------- |
| `00_概要.md`           | 关键洞察、文件导览、执行顺序                |
| `01_访谈记录.md`       | 访谈与材料收集                              |
| `02_原子拆解.md`       | 全部原子及输入 / 输出 / 成功标准 / 工具映射 |
| `03_评估与市面方案.md` | 全部原子的 8 维评估 + 市面方案搜索结果      |
| `04_AI-First重做.md`   | 演进版 + 重构版 + 红灯 + MVP                |

旧版多文件模板只作为字段参考，不按目录结构照搬。

## 启动流程（开场必做）

### 设计意图

用户说"拆我的工作"时，他的心理状态是"我想搞清楚"，不是"我准备填表"。启动流程的职责是：**用最少的摩擦让对话流动起来，同时让 AI 在后台完成所有系统准备**。

核心原则：
- 用户感知到的是"自然对话"，不是问卷
- AI 判断入口（不让用户选——90% 走 A）
- **不让用户描述工作流**——AI 基于上下文推理出来，用户只确认/修正
- 材料检测和快照处理是后台静默逻辑，不暴露给用户

### Step 0：暖机邀请（一句话开场）

**AI 只说一句话**，目的是让用户开口讲一件具体的事：

> 你想拆哪件事？随便说——叫什么、大概干嘛的、最近一次是什么时候做的。说多说少都行，我来帮你抠。

**变体**（根据触发方式选用）：

| 触发方式                      | 开场话术                                                                         |
| ----------------------------- | -------------------------------------------------------------------------------- |
| 用户说"拆我的工作"            | "好，挑一件你反复做的事——哪件最让你觉得'这玩意儿不应该还是这么做'？"             |
| 用户说"AI 能帮我到什么程度"   | "具体是哪件事？给我一件你最想知道 AI 能做到什么程度的活儿。"                     |
| 用户直接贴了 SOP / 文档       | 不问，直接读材料，走 C 路径追问                                                  |
| 用户说"摊开我自己" / "完整来" | "好，从你开始。你现在手里同时在做几件事？大概列一下，不用排序。" → 进 B 完整入口 |
| 用户带着 ≥3 份报告来          | 检测到已有报告 → 直接确认"你想把这些人组合起来看？" → 进 C 企业模式              |

**核心：不让用户选 ABC**。AI 根据用户第一句话判断入口：
- 说了一件具体的事 → A 轻量
- 说了"全面看看" / 列了多件事 → B 完整
- 带了多份已有报告 → C 企业

如果 AI 判断错了？没关系——随时可以切。

### Step 0.5：后台静默处理（用户不可见）

用户回答 Step 0 的同时，AI 后台做三件事：

1. **材料检测**：
   - 用户贴了链接/文件/SOP → 标记"有材料"，读完后走 C 路径
   - 用户只用自然语言描述 → 标记"无材料"，走 A 路径
   - 不需要问用户"你有没有材料"——他有就会贴

2. **快照状态检查**：
   - 读 [snapshots/ai-capability-snapshot.md](snapshots/ai-capability-snapshot.md) 的最后更新时间
   - 过期 > 7 天 → 内部标记 `snapshot_stale = true`，**不告诉用户**
   - 在后续评估阶段，所有方案自动加 `⚠️ 快照过期，方案推荐需补验证`

3. **入口路由**：
   - 根据用户第一段描述判断走 A/B/C，不显式告知用户

### Step 1：收集推理地基（3-5 轮对话）

用户回答了 Step 0（说了一件事），**AI 不说"好的我们开始访谈"**。直接接住他说的内容，用追问的方式收集推理工作流所需的信息。

**v4.0 核心变化**：不问"你的流程是什么"——问"你是谁、你做什么、你给谁东西"。AI 基于这些好回答的问题推理出工作流框架。

详见 [references/A路径访谈脚本.md](references/A路径访谈脚本.md) 的 Step 1 最小信息集。

**最小信息集**（7 项，够推理工作流框架）：

| #   | 信息                             | 为什么需要           |
| --- | -------------------------------- | -------------------- |
| 1   | 公司干什么的（行业、规模、阶段） | 行业决定工作流模式   |
| 2   | 他在什么部门/团队                | 部门决定职责边界     |
| 3   | 部门核心职责                     | 职责决定产出类型     |
| 4   | 上下游是谁                       | 上下游决定流程节点   |
| 5   | 评估标准                         | 标准决定原子的成功门 |
| 6   | 经常产出什么给别人               | 产出决定流程终点     |
| 7   | 触发条件                         | 触发决定流程起点     |

**对话节奏**：不宣布段落，从用户上一句自然接话，一次 1-2 题。3-5 轮收集完。

**结束信号**：7 项基本填满（允许 ≤1 项模糊）→ AI 进入 Step 2。

### Step 2：AI 推理工作流框架 + 用户确认修正

AI 基于 Step 1 信息推理出"这个人大概有哪些工作流"，每条拆到**步骤级**，输出给用户确认。

**步骤级 vs 原子级**：

|          | 步骤级                     | 原子级                                                   |
| -------- | -------------------------- | -------------------------------------------------------- |
| 粒度     | 一个可识别的阶段/动作块    | 不可再拆的最小执行单元（四门全过）                       |
| 例子     | "收集竞品信息"             | "打开XX数据库→输入关键词→筛选近30天→逐条阅读→摘录到文档" |
| 数量     | 一条工作流大概 4-8 步      | 一个步骤可能拆出 3-10 个原子                             |
| 判断标准 | 用户能说"对，我有这个阶段" | 能映射到一次工具调用/一次人判断                          |

**输出格式**：每条工作流列出步骤 + 标注推理依据。用户看着说"对/不对/少了/多了"。

**增补审视（v4.1 新增）**：推理出步骤后，AI 主动审视"这条流程里缺什么步骤"——不仅是用户漏说了，而是**原来因人力不够而省略、但 AI 介入后应该补回来的环节**。增补步骤单独标注 `[✚ 建议新增]`，和用户确认的现有步骤区分开。典型方向：质量校验、多版本对比、知识沉淀、事后复盘、预检/风险扫描。

**用户修正后**：AI 推荐先拆哪条（基于高频/最烦/最耗时），进入 Step 3。

### Step 3：选一条深入 + 补充个人信息

用户确认框架后，选一条先完整跑。同时收集后续方案设计需要的个人信息（2-3 轮）：

| #   | 信息                          | 后续对应              |
| --- | ----------------------------- | --------------------- |
| 8   | 职级/角色（执行者 vs 决策者） | 原子的"不可委托"标记  |
| 9   | 最常用工具                    | AI-First 重做的接入点 |
| 10  | 公司 AI 政策                  | 方案的可行边界        |
| 11  | AI 现有使用程度               | 演进版的起点          |
| 12  | 可用资源                      | 原子评估的约束        |
| 13  | 痛点（具体哪个环节最烦）      | 原子的痛点标记        |
| 14  | 改变动机                      | 方案的激进程度        |

**门禁**：8-14 项基本收集完 → 进入原子拆解（阶段 2-8）。

### Step 2 入口分支处理（AI 内部路由，对用户透明）

| AI 判断                          | 下一步                                                                                             |
| -------------------------------- | -------------------------------------------------------------------------------------------------- |
| A 轻量（用户聊的是一件具体的事） | 走 A 路径访谈脚本 Step 1-3，推理工作流框架                                                         |
| B 完整（用户想全面看）           | 走 [references/个人三层结构.md](references/个人三层结构.md)，先建人物画像                          |
| C 企业                           | 检查有 ≥3 份全人报告 → 走 [references/企业从个人组合方法论.md](references/企业从个人组合方法论.md) |

## 单条活儿审计流程 v1.2（步骤文件版）

适用于 A 入口、B 入口里的逐条审计、以及全人合成的中间产物。

**输出目录**：`02AI 余一/工作流审计/[活儿名]_[YYYY-MM-DD]/`

### 阶段 1：收集推理地基 + 推理工作流框架 + 用户确认 → 写入 `01_访谈记录.md`

- 有材料：走 [references/C路径追问清单.md](references/C路径追问清单.md)
- 无材料：走 [references/A路径访谈脚本.md](references/A路径访谈脚本.md)（v4.0）

**v4.0 核心流程**：

```
Step 1: 3-5 轮对话收集推理地基（7 项最小信息集）
Step 2: AI 推理工作流框架（步骤级），输出给用户确认/修正
Step 3: 选一条先拆 + 2-3 轮补充个人信息（职级/工具/AI政策/AI使用程度/痛点/动机）
```

**真对话约束硬执行**：一次只问 1-2 题，等真回答，再下一题。**不问"你的流程是什么"——AI 推理出来让用户确认**。

**推理填充模式（v1.2 保留）**：

当满足以下任一条件时，可启动推理填充模式替代逐轮对话：

1. 用户已在开场描述中覆盖了最小信息集 ≥5/7 项
2. 用户明确说"你直接拆吧"/"我说完了你来"/"不用再问了"
3. 上下文已有材料（SOP/文档/历史交付）能推理出最小信息集

**推理填充执行规则**：

```
Step 1: AI 基于已有信息推理 7 项地基 + 工作流框架
Step 2: 每项标注来源：[用户原话] / [AI 推理：因为...] / [待确认]
Step 3: 输出给用户确认/修正
Step 4: 用户确认后进入阶段 2
```

**物理门禁**：

- 7 项推理地基 + 工作流框架已用户确认 → 可以进阶段 2
- 个人信息（#8-14）基本收集完 → 可以开始原子拆解
- [待补] 字段 ≤ 2 项 → 可以进阶段 2，但需在 00 概要标注信息缺口

访谈完成后落盘 `01_访谈记录.md`（模板：[templates/单条审计文件系列/01_访谈记录.md](templates/单条审计文件系列/01_访谈记录.md)）。

### 阶段 2：原子拆解 → 写入 `02_原子拆解.md`

按 [references/原子拆解标准.md](references/原子拆解标准.md) 把活儿拆到原子级。**四门必过**（输入/输出/可评估/工具映射）。

输出：`02_原子拆解.md`（模板：[02_原子拆解.md](templates/单条审计文件系列/02_原子拆解.md)）。

### 阶段 3：派发 subagent 并行搜方案 → 汇总进 `03_评估与市面方案.md`

按 [references/搜索方案subagent调用规约.md](references/搜索方案subagent调用规约.md)。

**操作**：

1. 对每个原子构造 scout-agent 任务（用规约里的 prompt 模板）
2. **一次消息内并行派发全部任务**（不要串行）
3. 等全部返回
4. 验收每个产物（推荐+URL+日期+搜索过程+淘汰名单）
5. 不通过的重派
6. 按原子小节汇总写入 `03_评估与市面方案.md`，不创建 per-atom 文件

**参考**：评估开始前必读 [snapshots/ai-capability-snapshot.md](snapshots/ai-capability-snapshot.md)。

### 阶段 4：8 维评估 → 写入 `03_评估与市面方案.md`

按 [references/评估维度卡.md](references/评估维度卡.md)。

**评估分级标准（v1.2 新增）**：

原子数量多时（≥ 10 个），按以下分级执行，兼顾质量和可执行性：

| Tier                | 适用原子                                                | 评估深度            | 最小输出                                             |
| ------------------- | ------------------------------------------------------- | ------------------- | ---------------------------------------------------- |
| Tier 1（全展开）    | 痛点原子 + "部分能-需校验"原子 + 用户特别关注的         | 8 维全展开卡片      | 8 维 + 市面方案 + 成本 + 风险 + 验证步骤             |
| Tier 2（4 维+验证） | 非痛点但高频原子 + "部分能-辅助"原子                    | 4 维卡片 + 验证步骤 | 能力等级 + 人机分工 + 推荐路径 + 风险概要 + 验证步骤 |
| Tier 3（总览行）    | "能-可独立"/"超过人类"（已很好） + "不能"（直接进红灯） | 总览表一行          | 能力等级 + 人机分工 + 推荐路径 + 分级理由            |

**分级判断时机**：阶段 2 完成后、阶段 3 开始前。写入 `03_评估与市面方案.md` 开头的"评估分级表"。

**Tier 2 的 4 维如何生成（不是跳过，是结构化推理）**：
1. 从 Tier 1 原子的评估结果中提取模式（同类工具→相似能力）
2. 用模式外推 Tier 2 原子的 4 维
3. 每个 Tier 2 原子标注"推理依据：参考原子 X 的 Y 维度"
4. 验证步骤不能省——即使是推理生成的评估，也要给 5 分钟可跑的验证动作

**Tier 3 进红灯的规则**：
- "不能"的原子 → 自动写入 04 红灯清单（来源 A）
- "能-可独立"/"超过人类" → 不需要改造，标注"保持现状"

**输出一层**：`03_评估与市面方案.md` 先放评估分级表，再放总览表，再按 Tier 分区写评估。

### 阶段 5：AI First 重做（真方案）→ 写入 `04_AI-First重做.md`

不允许"建议……"模糊措辞。每条都要"具体动作 + 完成信号"。

### 阶段 6：红灯清单（跨路径）→ 写入 `04_AI-First重做.md`

## 递进全拆流程（第一条跑完后）

**设计理念**：不是"选择先拆一条还是全拆"，而是**递进**——第一条是校准标尺，后面 AI 基于标尺推理全部，用户统一修正。

### 触发条件

第一条工作流的 00-04 文件全部落盘完成。

### 执行流程

1. **AI 告知用户**：
   > "第一条拆完了。我现在按同样的标准把其他工作流也跑一遍，跑完给你统一看。"

2. **AI 自动跑全部**（不中间打断问用户）：
   - 对每条工作流走阶段 2-8 全流程
   - 以第一条的拆解粒度、评估深度、方案激进程度为标尺
   - 每条产出独立的 00-04 文件，落到各自的目录

3. **全部完成后输出总览**：

   ```
   全部跑完了。总览：

   | 工作流          | 原子数 | AI 能完全做 | 需要人 | 核心痛点 | AI-First 关键改动 |
   | --------------- | ------ | ----------- | ------ | -------- | ----------------- |
   | [第一条-已确认] | ...    | ...         | ...    | ...      | ...               |
   | [第二条]        | ...    | ...         | ...    | ...      | ...               |
   | ...             | ...    | ...         | ...    | ...      | ...               |

   哪条你觉得拆得不对、或者方案不合理？我来改。
   ```

4. **用户修正**：用户看完统一给反馈，AI 更新对应文件。

### 为什么不中间打断

- 用户已通过第一条校准了标准（粒度、深度、方案风格）
- 没看到全貌时给不出好反馈
- 中间打断只让流程拖长
- 拿到全部结果统一修正更高效
