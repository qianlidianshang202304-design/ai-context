# 市面方案搜索 SOP

> **本 skill 所有联网搜索通过 `.yuyi/skills/web-access` 完成。** 不允许裸 WebSearch。

宁少推、不过期。半年后回看不打脸是这个 skill 的命。

---

## 三道闸

### 闸 1：信息源选择

按 [市面方案信息源地图.md](市面方案信息源地图.md) 选源。先确定原子类型，再去对应的源。

### 闸 2：5 条硬门（候选过筛）

每个候选方案必须**同时**满足以下 5 条，否则不进推荐栏：

1. **时间门**：信息发布时间 ≤ **3 个月**（默认）。3 个月内候选 < 3 个 → 放宽到 6 个月，但必须标注"超 3 个月，已重新验证可访问性 + 最新版本"。超 6 个月一律砍。
2. **可访问门**：用户能注册、能调 API、能下载，不是"内测中""排队中""仅限企业咨询"。
3. **可验证门**：有公开 demo / 试用 / 开源代码 / 第三方评测之一。纯厂商自吹一律砍。
4. **匹配门**：方案的"输入-输出"和原子的"输入-输出"必须在同一颗粒度。例：原子是"从 30 条信息归类为 3-5 簇"，那"通用 RAG 框架"不算匹配，"聚类 + 主题命名"才算。
5. **替代门**：和已有推荐方案相比，必须在某个维度（成本 / 质量 / 速度 / 易用性 / 隐私）有 ≥30% 优势，否则只列不推。

### 闸 3：4 步检索动作

每个原子按这个顺序跑：

```
Step 1: 双语搜索
  - 调用 web-access 搜索：[动作英文关键词] 2026 best tool
  - 调用 web-access 搜索：[动作中文关键词] 工具 / 智能 / 自动化
  - 收 10-15 个候选，去重

Step 2: 候选验证
  - 对每个候选调用 web-access 抓取官方介绍页 + 1 个第三方评测
  - 套 5 条硬门过筛

Step 3: 推荐生成
  - 过门候选 ≥ 3 个：选最强 2 个推荐 + 列其他 ≤2 个备选
  - 过门 1-2 个：直接推
  - 过门 0 个：标注"暂无成熟方案"，进入"自建可能路径"分支

Step 4: 来源记录
  - 每个推荐必须附 URL + 访问日期（YYYY-MM-DD 格式）
  - 不附 = 视为无效推荐，必须重做
```

## "暂无成熟方案"的处理

某个原子搜不到能用的方案时，**这是宝贝**——意味着这是 AI First 重做的优先攻击点。

处理方式：
1. 在评估卡上明确标注"暂无成熟方案"
2. 给出"自建可能路径"：通用 LLM + 什么 Prompt 模式 / 什么 Agent 框架可以拼出来
3. 把这个原子标记为"AI First 红点"，进入路径 B 重构式重做时优先看

## 验收清单（每跑完一个原子的搜索必查）

- [ ] 至少 1 次中文搜索 + 1 次英文搜索
- [ ] 推荐方案在过去 3 个月内有更新（看 GitHub last commit / 产品更新日志 / 新闻发布日期）
- [ ] 每个推荐都有 URL + 访问日期
- [ ] "暂无成熟方案"的原子已标记并写入 AI First 红点清单
- [ ] 没有引用任何半年前的"神文"作为唯一证据

## 快照引用规则

评估前必查 [snapshots/ai-capability-snapshot.md](../snapshots/ai-capability-snapshot.md)：

| 快照状态 | 行动 |
|---|---|
| 上次更新 ≤ 7 天 | 直接引用 |
| 上次更新 8-14 天 | 引用 + 提示用户"快照已 X 天未更新" |
| 上次更新 > 14 天 | 强制运行刷新流程或在评估末尾标注"使用了 X 天前的快照，可能有偏差" |

## 反作弊

AI 在偷懒时常见的几种作弊方式，提前禁掉：

1. ❌ 凭训练记忆吐"我记得有个工具叫……" → 必须现搜
2. ❌ 列方案不列 URL → 视为不存在
3. ❌ 用 1 年前看过的产品当"最新方案" → 必须验证更新日期
4. ❌ 把"通用 LLM 都能做"当成方案 → 不是方案
5. ❌ 把厂商 PPT 上画的能力当成已交付方案 → 必须验证可访问门
