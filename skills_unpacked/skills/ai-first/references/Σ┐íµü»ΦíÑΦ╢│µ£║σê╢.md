# 信息补足机制（v1.2 新增）

> 核心原则：信息不足时**不跳过**，而是用多种方式让信息充足。
>
> "跳过"是 AI 偷懒；"只问用户"是把责任甩回去。真正的解法是 AI 主动补足，再让用户确认/纠正。

---

## 为什么需要这个机制

v1.0/v1.1 执行中暴露的核心问题：skill 设计了"理想态硬门"但没给"现实态降级链"——当环境不满足假设时，AI 只有两个选择：违规执行 or 整体卡住。两个都不是好结果。

信息补足机制的目标：**在不降低产出质量底线的前提下，让 AI 在信息不完美时仍然能产出有价值的东西，并且清楚标注哪些部分需要后续补验证。**

---

## 四层信息补足路径

| # | 路径名 | 做法 | 适用场景 | 关键约束 |
|---|---|---|---|---|
| 1 | 推理填充 + 用户确认 | AI 基于已有信息推理生成完整内容，用户只改不对的 | 用户给了足够背景但没逐条回答 | 必须标注 [AI 推理]，必须经用户确认 |
| 2 | 结构化推理（同类外推） | 用已知原子的评估结果/模式，外推到未知原子 | 同一工作流内相似原子的评估 | 必须标注"推理依据：参考原子 X"，验证步骤不可省 |
| 3 | 模型知识 + 可验证标记 | 用模型知识生成初稿，明确标为待验证，给验证动作 | 搜索不可用时的市面方案 | 禁止写 URL，必须附验证步骤和搜索词 |
| 4 | 分阶段交付 | 第一轮标明信息缺口，后续补充时触发补全 | 信息无法在当次获得 | 必须在概要标注缺口 + 对后续评估的具体影响 |

---

## 路径 1：推理填充 + 用户确认

### 适用场景

- 访谈阶段：用户已在开场描述中覆盖了大部分信息
- 评估阶段：从访谈内容可以推理出某些评估维度的取值

### 执行规则

```
1. AI 根据已有信息生成完整内容
2. 每个字段标注信息来源：
   - [用户原话]：用户直接说过的，引用原文
   - [AI 推理]：从上下文逻辑推导的
     - 必须附一句话推理依据（"因为你说了 X，我推理 Y"）
   - [待确认]：信息不足无法推理，标出不确定点
3. 输出给用户确认
4. 用户确认后视为合法输入
```

### 与"自填字段"的区别

| 自填字段（违规） | 推理填充（合规） |
|---|---|
| 不标注来源 | 每字段标注来源 |
| 不让用户确认 | 必须经用户确认 |
| 假装是用户说的 | 明确是 AI 推理的 |
| 跳过后续步骤 | 不影响后续执行 |

---

## 路径 2：结构化推理（同类外推）

### 适用场景

- 评估阶段：Tier 1 原子已有完整评估，Tier 2 原子可以外推
- 同类原子有相似的输入/输出/工具映射

### 执行规则

```
1. 识别原子之间的结构相似性（同类工具、同类输入、同类判断）
2. 从已完成 8 维评估的原子中提取模式
3. 用模式外推：
   - "原子 A 用 web-search 做信息收集，评估为'部分能-需校验'"
   - "原子 B 也用 web-search 做信息收集，结构相似"
   - → 外推原子 B 为"部分能-需校验"，依据：与原子 A 结构相似
4. 标注推理依据："推理依据：参考原子 {N} 的 {维度}"
5. 验证步骤仍然必须给——可以写"参考原子 {N} 的验证步骤，替换输入为 {本原子的输入}"
```

### 质量保证

- 外推只适用于结构相似度 > 70% 的原子
- 如果两个原子虽然工具相似但判断维度差异大（如一个有合规风险一个没有），不能外推风险维度
- 宁可写"Tier 2 评估，推理生成"也不能写成看起来像独立评估的样子

---

## 路径 3：模型知识 + 可验证标记

### 适用场景

- 市面方案搜索不可用（L3 模式）
- 某些评估维度需要外部信息但无法获取

### 执行规则

```
1. 用模型已有知识生成内容
2. 显著标注"模型推理生成，未经搜索验证"
3. 禁止写 URL（URL 是搜索证据，推理不产生证据）
4. 每个推理内容必须附验证步骤：
   - 具体搜索关键词（中英文各 2-3 组）
   - 验证方法（去哪搜、看什么）
   - 判断标准（搜到什么算确认、搜到什么算推翻）
5. 在 00 概要和 04 红灯中显著标记
```

### 与"凭记忆补位"的本质区别

| 凭记忆补位（违规） | 模型知识+可验证标记（合规） |
|---|---|
| 写 URL 伪装搜索结果 | 禁止写 URL |
| 加 ⚠️ 当脚注 | 验证变成用户第一个可执行动作 |
| 概要不标注 | 概要显著标注执行模式 |
| 红灯不记录 | 红灯增加"搜索未执行"条目 |
| 读者可能忽略 | MVP 第一步就是补验证 |

---

## 路径 4：分阶段交付

### 适用场景

- 信息只能在下次对话/用户补充后才能获得
- 访谈追问后仍有字段标 [待补]
- 需要外部事件触发（如快照刷新）

### 执行规则

```
1. 第一轮交付时：
   - 完成能完成的部分（有足够信息的原子正常评估）
   - 信息缺口部分写"待补：[具体缺什么]"
   - 00 概要标注"本版缺口清单 + 对评估的具体影响"
   - 在 MVP 中加"补全信息"动作

2. 后续补充时：
   - 用户提供新信息后，直接补全对应区域
   - 补全后重新过自检清单对应项
   - 概要更新为"完整版"
```

### 分阶段 vs 跳过 的区别

| 跳过（违规） | 分阶段交付（合规） |
|---|---|
| 不提缺口 | 显著标注缺口 |
| 不说影响 | 标明对后续哪些环节有影响 |
| 假装完整 | 明确标注版本状态 |
| 不给补全路径 | MVP 包含补全动作 |

## 四条路径的选择决策树

```
信息不足时：
│
├─ 能从上下文推理吗？
│  ├─ 能（>70% 确信）→ 路径 1（推理填充 + 用户确认）
│  └─ 不能
│     │
│     ├─ 有同类原子的已知评估可外推吗？
│     │  ├─ 有（结构相似>70%）→ 路径 2（结构化推理）
│     │  └─ 没有
│     │     │
│     │     ├─ 模型知识能给出有意义的初稿吗？
│     │     │  ├─ 能 → 路径 3（模型知识 + 可验证标记）
│     │     │  └─ 不能 → 路径 4（分阶段交付，标缺口）
│     │     │
│     │     └─ 补充：用户在场且问题可快速回答？
│     │        └─ 是 → 追问用户（但限 1-2 次，不死缠）
│     │
│     └─ 补充：如果是 Tier 3 原子（评估要求低）
│        └─ 直接给总览行 + 分级理由，不需要补足
│
└─ 绝对禁止：
   - 不标注就跳过
   - 伪装为完整信息
   - 无限追问用户（> 2 次同一问题）
```

---

## 与 skill 各阶段的对应

| 阶段 | 信息不足常见场景 | 推荐路径 |
|---|---|---|
| 阶段 1 访谈 | 用户描述不完整 | 路径 1（推理填充）→ 追问 → 路径 4（标 [待补]） |
| 阶段 2 原子拆解 | 不确定某步骤的工具映射 | 路径 1（推理 + 确认） |
| 阶段 3 市面方案 | 搜索不可用 | 路径 3（模型知识 + 可验证标记） |
| 阶段 4 评估 | 原子太多无法全展开 | 路径 2（同类外推）+ 分级标准 |
| 阶段 5 AI First | 不确定 AI 当前能力边界 | 路径 3 + 路径 4（标红灯 + 验证步骤） |
