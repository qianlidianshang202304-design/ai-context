# 搜索方案 subagent 调用规约（v1.0 新增）

> 市面方案搜索必须并行，不能串行。10 原子 × 5 调用 = 50 次串行 = skill 跑不动。

## 基本原则

1. **每个原子一个 subagent 任务**：原子之间相互独立，并行最高效。
2. **使用 `scout-agent`**（角色路由：信息搜集者）。理由：工作区已有该角色，内置 web-access 能力，专做信息搜索。
3. **不强绑定具体模型**：v1.0 留出"模型偏好可配置"的口子（want-to-do）。当前默认走主调度模型。
4. **每个 subagent 任务输入必须自洽**：原子描述 + 工具/Prompt 映射 + 信息源类型偏好（参考 [市面方案信息源地图.md](市面方案信息源地图.md)）。subagent 不读对话历史。
5. **统一输出格式**：每个 subagent 必须按下面的"输出契约"返回。

## 调用流程

```
原子拆解完成（02_原子拆解.md 落盘）
  ↓
对每个原子构造一个 subagent 任务（见下面"任务 prompt 模板"）
  ↓
并行派发 N 个 scout-agent 任务（一次消息发出 N 个 Task 调用）
  ↓
等待全部返回
  ↓
按"输出契约"校验每个 subagent 的产物
  ↓
逐原子整理搜索结果
  ↓
汇总写入 03_评估与市面方案.md 的对应原子小节
```

## 任务 prompt 模板（派给 scout-agent）

```
[任务] 为以下原子搜集市面最新方案。

[原子描述]
- 名称：{原子名}
- 输入：{输入描述}
- 输出：{输出描述}
- 成功标准：{标准}
- 工具/Prompt 映射：{已有的初步映射}

[搜索范围]
- 时间窗：默认近 3 个月，必要时扩到 6 个月
- 信息源优先级（参考工作区 .yuyi/skills/ai-first-redesign/references/市面方案信息源地图.md）：
  - 通用 LLM 能力 → OpenAI / Anthropic / Google 官方文档 + 官方博客
  - 垂直 AI 工具 → 工具官网 + Product Hunt + Twitter/X 实测帖
  - Agent / 工作流 → LangChain / LlamaIndex / n8n / Dify 等开源 + 评测博客
  - 国内特化 → 智谱 / 通义 / 豆包 / Kimi / 月之暗面等
- 双语必做：中英文各搜一轮
- 来源筛选：官方 > 第三方实测 > 媒体宣传

[硬筛标准]
1. 时间：发布/更新日期在时间窗内
2. 真实性：能找到 demo/截图/代码/复现说明，不是纯营销文案
3. 相关性：直接对应"工具/Prompt 映射"，不是擦边球
4. 可达性：免费可试 / 有 API / 有开源版本
5. 来源可信：官方文档优先，独立开发者次之，营销博客最低

[输出契约]
按以下 markdown 格式返回，至少 1 推荐 + ≤2 备选：

## 原子 {N}：{原子名} - 市面方案

### 推荐
- **方案名**：{产品/工具/框架/Prompt 模式}
- **提供方**：{厂商/开源/独立}
- **能力**：{一句话}
- **适用边界**：{什么场景能用，什么不行}
- **上手成本**：{API 调用 / 注册即用 / 自建 / 采购}
- **来源**：{URL} @ {YYYY-MM-DD}
- **筛选理由**：{为什么从搜索结果里选这个}

### 备选 1（如有）
…

### 备选 2（如有）
…

### 搜索过程
- 中文搜索词：{用了哪些}
- 英文搜索词：{用了哪些}
- 命中但被淘汰的方案：{方案名 + 淘汰原因}（≥1 条，证明搜过对比过）

[禁止]
- 不允许编造 URL
- 不允许"据说"/"听闻"/"可能"等模糊措辞
- 不允许只给推荐不给搜索过程
- 找不到合格方案就明确写"未找到符合标准的方案"——这是合理结果，会触发红灯
```

## 验收清单（主 agent 收到 subagent 产物后必检）

- [ ] 至少 1 个推荐 + 完整字段（含 URL + 日期）
- [ ] 搜索过程包含中英文搜索词
- [ ] 至少 1 条"命中但淘汰"的方案，证明做了对比
- [ ] URL 可点开（抽检 2-3 条）
- [ ] 时间在窗内
- [ ] 没有"据说""可能"等模糊措辞

任一项不通过 → 重新派发 subagent。

## "未找到方案" = 红灯触发

如果某原子 subagent 返回"未找到符合标准的方案"——
- 不要再纠结
- 直接把该原子记入 `04_AI-First重做.md` 的红灯清单，来源标注 A（演进路径上 AI 还做不到）
- 主 agent 在 `03_评估与市面方案.md` 的对应原子小节写："未找到（详见 04_AI-First重做.md 红灯清单）"

## 性能指标（参考）

- 一个 scout-agent 任务的工作量：单原子 1 次任务派发 + 1 次完成回调
- 并行度：取决于运行环境。建议一次发出全部原子的搜索任务（一次消息内多个 Task 调用）
- 超时处理：单个 subagent 失败/超时不阻塞其他，最后统一 retry 失败的

## 环境降级链（v1.2 新增）

> 当运行环境不满足"并行 subagent 派发"的理想条件时，按以下优先级逐级降级。
> 降级不是违规——是在标注约束下的合规替代。降级 ≠ 跳过。

### 降级判断流程

```
能并行派发 subagent 吗？
├─ 能 → L1（理想态），按上方规约执行
└─ 不能
   ├─ 有 web-access 能力吗？
   │  ├─ 有 → L2（主 agent 串行搜索 Tier 1 原子）
   │  └─ 没有 → L3（模型知识 + 可验证标记）
   └─ 注意：在判断"能/不能"时，必须实际尝试派发 1 个 subagent
          只凭推测"可能不行"不允许直接降到 L3
```

### L2：主 agent 串行搜索

- 适用：subagent 不可用但有 web-access
- 做法：
  1. 对 Tier 1 原子（痛点区，≤ 8 个）逐个使用 web-access 搜索
  2. 搜索词和筛选标准同上方"任务 prompt 模板"
  3. 非 Tier 1 原子 → 走 L3
- 输出格式：同上方"输出契约"，但标注"L2 模式执行：主 agent 串行搜索"
- 缺点：慢，但搜索质量不降级

### L3：模型知识 + 可验证标记

- 适用：完全无法联网搜索
- 做法：
  1. 市面方案区标题写"**模型推理生成，未经搜索验证**"
  2. **禁止写 URL**——URL 是搜索证据，推理不产生证据
  3. 可以写方案名 + 能力描述（模型知识范围内）
  4. 每个方案必须附"验证步骤"：

     ```
     ### 验证步骤（用户/AI 补跑时执行）
     - 搜索关键词（中文）：[2-3 组]
     - 搜索关键词（英文）：[2-3 组]
     - 验证方法：[去哪搜、看什么页面]
     - 确认标准：[搜到什么算方案存在且可用]
     - 推翻标准：[搜到什么算方案已废弃/不适用]
     ```

  5. `04_AI-First重做.md` 红灯清单增加："方案搜索未执行（L3 模式）"
  6. MVP 第一步改为"补跑方案验证"
  7. `00_概要.md` 显著标注："本次评估在 L3 模式下完成，市面方案为推理生成"

### L3 与违规的对照

| 凭记忆补位 + 标 ⚠️（违规） | L3 合规替代 |
|---|---|
| 写 URL | 禁止写 URL |
| ⚠️ 标记是脚注 | "模型推理生成"是标题级标注 |
| 不影响后续流程 | 红灯 + MVP 第一步强制补验证 |
| 概要不标注 | 概要显著标注执行模式 |
| 读者可能忽略 | 验证是可执行动作，不是提醒 |
