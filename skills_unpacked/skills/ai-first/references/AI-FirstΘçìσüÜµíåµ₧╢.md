# AI-First 重做框架（v1.1）

> v1.1 要求：演进版、重构版、红灯、MVP 全部写入 `04_AI-First重做.md`。AI First 只有一个文件入口。
> 红灯方法见 [AI-First红灯清单方法论.md](AI-First红灯清单方法论.md)。

## 两条路径

### 路径 A：演进式（Evolution）

> 在原子拆解的基础上，对现有流程做"消、并、改、调"。

**演进五问**（每个原子都过一遍）：
1. **消**：这步能不能直接被 AI 替代后人不再参与？
2. **并**：这步能不能和上下文原子合并成一个 AI 可一次完成的更大原子？
3. **改**：这步如果交给 AI 主导，工具/Prompt/数据准备需要怎么改？
4. **调**：这步在流程里的位置是否需要前移/后移/并行化（用 AI 做了之后顺序可能变）？
5. **增**：AI 介入后，这条流程里是否**缺少了本该有但因人力成本太高而省略的步骤**？

> **"增"的审查视角**：不是为了加步骤而加，而是回答——"如果这个环节几乎零成本，我会不会早就加了？" 典型增补方向：
> - 质量校验层（原来没人力做 QA → AI 低成本加一道自动校验）
> - 多版本/多方案对比（原来只出一版 → AI 能快速出 3 版让人挑）
> - 数据沉淀/知识留存（原来做完就丢 → AI 顺手结构化存档）
> - 事后复盘/模式提取（原来没空回头看 → AI 自动总结 pattern）
> - 上下游同步/通知（原来靠人记得通知 → AI 自动触发流转）
> - 预检/风险扫描（原来直接干 → AI 先跑一遍预检再干）

**输出要求**（写入 `04_AI-First重做.md` 的“路径 A：演进版”）：

1. **完整流程图（mermaid）**
   - 标出每个原子在新流程里的位置
   - 标出哪些原子"消失"（用虚线/灰色）
   - 标出哪些原子合并（用合并框）
   - 标出哪些原子改为 AI 主导（颜色编码）
   - 标出**新增步骤**（用绿色/实线框 + ✚ 标记），注明"原流程缺失，AI 低成本补回"

2. **每步工具映射表**

   | 原子 | 演进后状态 | 主工具 | Prompt/SOP | 输入数据准备 | 人介入点 |
   |---|---|---|---|---|---|
   | … | 消失 / 合并 / AI 主 / 人主 / **✚ 新增** | … | … | … | … |

3. **第一周实施 5 步**（具体动作，不抽象）
   - 步骤 1：[做什么][产出什么][谁做]
   - 步骤 2：…
   - …
   - 步骤 5：[第一周结束应得到什么状态]

4. **演进版关键指标**
   - 自动化率：原 X% → 演进后 Y%
   - 人介入次数：原 N 次 → 演进后 M 次
   - 关键质量指标：[活儿特定的]

### 路径 B：目标重构式（Reconstruction）

> 完全无视原流程。从目标和最终消费者倒推：如果今天从零开始，借助现在的 AI，最佳路径是什么？

**重构四问链**：
1. **真目标**：这件事的最终消费者是谁？他们真正想要的是什么？（不是当前交付物，是交付物背后想解决的问题）
2. **AI 视角**：如果一个有 100% 当前 AI 能力 + 完整工具链的智能体来做，它会怎么做？
3. **最少必要人**：在那个 AI 主导版本里，人无法被替代的环节是什么？为什么？
4. **关键缺口**：要让这个新流程跑通，AI 当前差什么？（→ 触发红灯来源 B）

**输出要求**（写入 `04_AI-First重做.md` 的“路径 B：重构版”）：

1. **目标重新定义**
   - 旧目标：[原本说的交付物]
   - 真目标：[消费者真正想要的]
   - 差距说明：[旧目标如何遮蔽了真目标]

2. **新架构图（mermaid）**
   - 关键模块：[列出所有模块]
   - 数据流：[模块间怎么传递]
   - AI/人/自动化标注：[每个模块谁来做]

3. **关键模块设计**（每个核心模块一段）

   ```
   ### 模块 N：[名称]
   - 职责：…
   - 输入：…
   - 输出：…
   - 实现方式：[AI 模型 / Agent / 自动化脚本 / 人]
   - 关键 Prompt 或 SOP：…
   - 与其他模块的接口：…
   ```

4. **迁移路径分阶段**

   ```
   ### 第一阶段（基础设施）
   - 目标：…
   - 关键动作：…
   - 完成信号：…

   ### 第二阶段（核心能力切换）
   - 目标：…
   - …

   ### 第三阶段（全面替代）
   - 目标：…
   - …
   ```

5. **关键缺口标记**
   - 在新架构图里把"AI 当前还差一口气"的节点标 ⚠️
   - 同步写入 `04_AI-First重做.md` 的“红灯清单”，来源标 B

6. **重构版关键指标**
   - 自动化率：原 X% → 重构后 Y%（应显著高于演进版）
   - 时间杠杆：原一次执行 [人手主导] → 重构后 [AI 自动 + 抽检]
   - 质量天花板：原 [熟练档] → 重构后 [资深档/专家档/超过人类]

## 三档颗粒度（每条路径都要给）

### 动作级（micro）
- 单个原子内部怎么改：换工具、换 Prompt、加数据预处理
- 例：原子"写小标题"从"人凭灵感"改为"AI 出 10 个 + 人选 3 个"

### 流程级（meso）
- 原子之间的关系怎么改：合并、并行、消失、新增
- **新增不只是"拆细"**——而是补回原来因人力不够而省略的环节（来自演进五问的"增"）
- 例 1：原"先大纲后写"改为"AI 同时出大纲和片段，人选片段反推大纲"
- 例 2：原"写完直接发"改为"写完 → AI 自动跑质量校验 + 风格一致性检查 → 人过一眼 → 发"

### 目标级（macro）
- 整件事的目标和交付物是否需要重新定义
- 例：原"写一篇 5000 字公众号长文"重定义为"产出一组 3 篇短卡片 + 1 个 5 分钟语音"

**强制要求**：路径 A 至少给到动作级 + 流程级；路径 B 必须给到流程级 + 目标级。

## 真方案 vs 建议清单（v1.0 硬区分）

### 不算真方案（v0.4 的水平）

```
建议：用 AI 写初稿，人来改稿。
```

### 算真方案（v1.0 要求的水平）

```
### 模块：初稿生成
- 职责：把大纲转成 5000 字的初稿，风格符合余一
- 输入：人定的大纲（含每节小标题 + 每节核心论点）+ 5 篇余一近期文（风格样本）
- 输出：5000 字 markdown，结构对齐大纲，风格 ≥ 熟练写作者一稿水平
- 实现：
  - 模型：Claude Opus（审美最稳）+ 备选 GPT-5
  - Prompt：《余一公众号写作风格守卫 v1》（路径见 modules/yuyi-style/）
  - 喂料策略：每次喂大纲 + 风格样本 5 篇 + 当前要写的小节
  - 迭代：分节生成不一次到 5000，避免 attention 散
- 接口：
  - 输入文件：00_提纲.md / 01_风格样本.md
  - 输出文件：02_初稿.md（按节切分）
- 与其他模块：
  - 上游：模块"大纲打磨"
  - 下游：模块"段间过渡"+ 模块"金句植入"
- 验证标准：
  - 朋友盲读 5 段，错认 ≥ 2 次（参考 [评估维度卡](评估维度卡.md) 验证步骤示例 1）
```
