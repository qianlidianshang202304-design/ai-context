# P01 - 单条活儿AI审计 - v1

## 状态

✅ · 最后验证日期：首次固化（基于 skill 文档反推） · 验证者：余一

## 适用判断

**用这份 practice，当且仅当：**
- 产出物：**5 个步骤文件**（00_概要 / 01_访谈记录 / 02_原子拆解 / 03_评估与市面方案 / 04_AI-First重做），落在 `02AI 余一/工作流审计/[活儿名]_[日期]/`
- 受众/场景：余一想拆解自己某条反复做的活儿，评估 AI 能做到什么程度，拿到真方案
- 平台/载体：本地 Markdown 文件系列
- 原始输入：一条具体的活儿描述 + 可选的已有文档/SOP/流程图

**不适用：**
- 完整人物画像 + 全人 AI First 合成 → 走 B 入口（`references/个人三层结构.md`）
- 企业/团队级全景 → 走 C 入口（需 ≥3 份全人报告）
- 只想了解某个 AI 工具能不能做某事 → 走 `web-access` 搜索
- 想要写 SOP/文档 → 走 `content-production`

## 输入契约

- [ ] 活儿名称（简洁可识别，如"每周内参生产"、"客户访谈整理"）
- [ ] 入口确认：选 A 轻量入口（本 practice 对应）
- [ ] 现成材料状态：有/无/部分有
- [ ] AI 能力快照状态：刷新/用现有+标注/未选

缺任何一项，停下来问余一，不要脑补。入口选择和快照状态在 Step 0 一起确认。

## 执行链（严格按顺序）

### Step 0 · 启动确认
- 调用：SKILL.md 启动流程
- 分叉/参数：
  - 入口选择：A 轻量（本 practice）/ B 完整 / C 企业
  - 现成材料：有 → 贴路径；无 → 对话访谈；部分 → 先贴有的
  - 快照：刷新 → 走 `hooks/weekly-snapshot-refresh.md`；未校准 → 所有方案加 ⚠️
- 通过标准：入口=A，材料状态和快照状态已确认
- 如果失败：余一犹豫 → 默认 A，先跑通一条再说

### Step 1 · 访谈与材料收集 → 写入 `01_访谈记录.md`
- 调用：`references/A路径访谈脚本.md`（无材料）/ `references/C路径追问清单.md`（有材料）
- 分叉/参数：一次只问 1-2 题，等真回答，再下一题
- 通过标准：访谈完成，`01_访谈记录.md` 落盘（模板：`templates/单条审计文件系列/01_访谈记录.md`）
- 如果失败：AI 自填字段 → 违反硬约束，必须删掉重来等用户真回答

### Step 2 · 原子拆解 → 写入 `02_原子拆解.md`
- 调用：`references/原子拆解标准.md`
- 分叉/参数：四门必过——输入/输出/可评估/工具映射
- 通过标准：每个原子有明确的输入、输出、成功标准、工具映射
- 如果失败：原子拆太细（按子文件） → 合并到步骤级

### Step 3 · 并行搜索市面方案 + 8 维评估 → 写入 `03_评估与市面方案.md`
- 调用：`references/搜索方案subagent调用规约.md` + `references/评估维度卡.md`
- 分叉/参数：
  - 对每个原子构造 scout-agent 任务
  - **一次消息内并行派发**全部任务（禁止串行）
  - 8 维评估：AI能力等级 / 预期质量 / 人机分工 / 市面方案 / 推荐路径 / 成本 / 风险 / 验证步骤
  - 第 8 维验证步骤必须 5 分钟可跑
- 通过标准：每个原子在 `03_评估与市面方案.md` 中有评估和方案，方案有 URL+日期+搜索过程+淘汰名单
- 如果失败：方案无 URL → 重派 subagent；串行调用 → 改为并行

### Step 4 · AI-First 重做 + 红灯 + MVP → 写入 `04_AI-First重做.md`
- 调用：`references/AI-First重做框架.md` + `references/AI-First红灯清单方法论.md`
- 分叉/参数：
  - 演进版：完整流程图(mermaid) + 工具映射表 + 第一周 5 步 + 关键指标对比
  - 重构版：真目标重定义 + 新架构图(mermaid) + 关键模块设计 + 迁移路径分阶段 + 关键缺口标记
  - 红灯清单：缺口描述 + 关键性 + 自建路径 + 等待信号 + 临时绕道
  - MVP：3-5 个动作，每个含 谁做/做什么/产出/完成信号/可验证
- 通过标准：演进版和重构版都完整；红灯和 MVP 作为一级小节写在同一文件
- 如果失败：只给演进版没给重构版 → 违反硬约束，必须补重构版

### Step 5 · 写概要 + 自检 → 写入 `00_概要.md`
- 调用：`templates/单条审计文件系列/00_概要.md` + `references/自检清单.md`
- 分叉/参数：≤5 条关键洞察 + A vs B 对比表 + 文件导览
- 通过标准：00_概要是最后写的，汇集 01-04 核心结论；自检全部通过
- 如果失败：自检发现时间预估词 → 全部改结构性指标

## 验收清单

- [ ] 5 个步骤文件全部落盘（00-04）
- [ ] 00_概要.md 是最后写的
- [ ] 01_访谈记录.md 来自真对话，非 AI 自填
- [ ] 02_原子拆解.md 每个原子四门必过
- [ ] 03_评估与市面方案.md 每个原子有 8 维评估 + 市面方案（含 URL+日期）
- [ ] 04_AI-First重做.md 含演进版+重构版+红灯+MVP
- [ ] 全文 grep 不到时间预估词（"约 X 分钟"等）
- [ ] 第 8 维验证步骤 5 分钟可跑，不是"试用看看"

任何一项未过，不要交付，回到对应 step 修。

## 已知失败模式

- ❌ 一次抛全部访谈问题 → 违反真对话约束，必须一次 1-2 题等真回答
- ❌ subagent 串行调用 10 次 → skill 跑不动，必须一次并行派发
- ❌ 只给演进版不给重构版 → 违反硬约束，必须有激进重构方案
- ❌ 市面方案凭记忆吐无 URL → 必须有搜索过程和淘汰名单
