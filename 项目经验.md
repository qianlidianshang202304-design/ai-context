## 📝 项目详情

### 项目一：内容AI工厂

- **时间**: 2025 ~ 至今
- **我的角色**: 全栈开发
- **团队规模**: 1 人 [推测]
- **技术栈**: 后端 FastAPI + SQLAlchemy + PostgreSQL + ChromaDB；前端 Next.js 16 + React 19 + TypeScript + Tailwind CSS + Zustand + Recharts
- **项目简介**: AI 驱动的内容创作和竞品分析平台，集成热点监控、竞品广告监控（Meta Ad Library）、素材管理、品牌视频分析、Reddit 洞察、AI 脚本生成、飞书同步等功能。

#### 我负责的部分
- 后端全部 API 设计（FastAPI Router 分层）
- 前端全部页面和组件（Next.js App Router）
- 竞品广告数据从 Meta Ad Library 的采集和存储（Apify + PostgreSQL）
- ChromaDB 向量检索用于素材相似搜索
- 飞书消息通知和多维表格同步集成
- Claude/Gemini API 用于 AI 内容生成和脚本创作

#### 技术难点 & 解决方案
- **竞品广告批量导入**：通过 Apify 调用 Meta Ad Library API，处理分页和限流，用 ChromaDB 做向量索引实现素材去重和相似搜索
- **Reddit 洞察**：爬取 Reddit 相关子版块帖子，用 AI 做情感分析和需求提取，结构化入库
- **前后端分离的数据一致性**：用 Zustand 管理前端状态，API 层统一错误处理（axios 拦截器）

#### 踩坑记录
- ChromaDB 数据持久化路径问题：Docker 挂载卷配置不对导致重启后数据丢失，需确保 `chroma-data/` 目录正确映射
- Meta Ad Library 数据字段不稳定：不同国家/地区的广告数据字段有差异，需要做兼容映射

#### 收获 & 思考
- 向量检索在素材管理场景非常实用，比传统关键词搜索效果好很多
- Zustand 比 Redux 简洁太多，适合中小型项目
- 飞书作为业务中台极大降低了内部工具开发成本

### 项目二：红人邮件回复系统（WonderBiotics KOL CRM）

- **时间**: 2025 ~ 至今
- **我的角色**: 全栈开发
- **团队规模**: 1 人 [推测]
- **技术栈**: Python FastAPI + SQLAlchemy + aiosqlite + Jinja2 模板 + Claude API + 飞书 SDK
- **项目简介**: 为 WonderBiotics 品牌构建的 KOL 邮件建联自动化 CRM。三栏布局工作台（左：收件箱 / 中：邮件正文+AI 建议 / 右：达人信息面板），支持多邮箱账号、批量发送（带像素追踪）、里程碑自动推进（M01→M23）。

#### 我负责的部分
- 全部后端 API（96KB 的 `routes/api.py`）
- 前端 Jinja2 模板渲染（SSR 风格）
- AI 辅助回信功能（Claude API 根据达人信息生成个性化回复建议）
- 邮件追踪像素实现（1x1 透明图片 + 唯一 ID）
- 飞书多维表格同步（达人状态回写）
- 多邮箱 IMAP 收件和 SMTP 发件管理

#### 技术难点 & 解决方案
- **多邮箱并发管理**：用 aiosqlite 存储多账号凭证，asyncio 并发收取多邮箱，避免阻塞
- **邮件追踪**：嵌入 1x1 透明像素图片，URL 带唯一 tracking ID，后端记录打开时间
- **AI 回信上下文**：将达人 Bio、历史邮件、品牌信息拼成 prompt，让 Claude 生成个性化回复
- **里程碑自动推进**：解析邮件内容关键词和用户手动标记，自动推进 M01→M02→...→M23

#### 踩坑记录
- aiosqlite 并发写入冲突：高频邮件收件时偶尔出现 database locked，加了重试机制和写入队列解决
- 飞书多维表格 API 限流：批量同步时触发限流，需要加延时和分批处理

#### 收获 & 思考
- aiosqlite 适合本地单机场景，但并发高时还是 PostgreSQL 更稳
- Jinja2 做 SSR 虽然传统，但对于这种内部工具来说开发效率高，不需要前后端分离
- AI 辅助回信极大提升了建联效率，但需要人工最终确认（Human-in-the-loop）

### 项目三：达人库分析系统

- **时间**: 2025 ~ 至今
- **我的角色**: 数据分析 + 脚本开发
- **团队规模**: 1 人 [推测]
- **技术栈**: Python/pandas + openpyxl + Instagram API + TikTok API + Modash 数据
- **项目简介**: 多数据源达人聚合分析系统。从 Modash、Instagram、TikTok 采集达人数据，进行 Bio 分类、邮箱补全、聚类分析、背书策略分析，输出营销报告和建联分层表。服务 IM8 / WonderBiotics 品牌。

#### 我负责的部分
- 20+ 数据处理脚本编写
- 达人 Bio 分类算法（AI 驱动）
- 缺失邮箱补全（API + 规则匹配）
- 聚类分析和分层策略
- Excel 报告自动生成

#### 技术难点 & 解决方案
- **多源数据去重**：Instagram、TikTok、Modash 数据源的达人 ID 体系不同，通过用户名+Bio 相似度做模糊匹配去重
- **Bio 分类**：用 Claude API 对达人 Bio 进行自动分类（Fitness / Lifestyle / Beauty 等），处理多语言 Bio
- **邮箱补全**：先尝试 API 查询，失败后用规则推断（名+姓+域名组合）

#### 踩坑记录
- Excel 大文件（10万+行）用 openpyxl 写入极慢，改用 xlsxwriter 或用 CSV 中间格式
- API 调用费用控制：Claude API 批量处理 Bio 分类时容易超预算，需要做缓存（已处理过的达人跳过）

#### 收获 & 思考
- pandas 的 `apply()` 配合 AI API 调用可以做很强大的数据增强
- 数据清洗脚本要考虑幂等性（重复运行不出错）
- 分层报告比原始数据表格对业务更有价值

### 项目四：处理FB文件

- **时间**: 2025
- **我的角色**: 脚本开发
- **团队规模**: 1 人
- **技术栈**: Python/pandas + Brave Search API + GPT API + openpyxl
- **项目简介**: Facebook 链接数据清洗补全工具。从 Excel 读取 FB 个人主页链接，调用 Brave Search 获取主页信息，GPT 做分类提取，输出标注后的 Excel 文件。

#### 我负责的部分
- 全部脚本开发（`process_fb.py`、`process_fb_hybrid.py`、`normalize_categories.py`）

#### 技术难点 & 解决方案
- **FB 页面信息抓取**：Facebook 反爬严格，改用 Brave Search API 间接获取页面摘要信息
- **分类归一化**：GPT 返回的分类名称不统一，写了 `normalize_categories.py` 做标准化映射

#### 踩坑记录
- Brave Search API 免费额度有限，大量链接需要分批处理和缓存
- GPT 分类偶尔不稳定，同样输入不同时间返回不同结果，需要做结果校验

#### 收获 & 思考
- 搜索引擎 API 是绕过社交媒体反爬的有效手段
- AI 做数据标注时要加校验层，不能完全信任 AI 输出

## 🔥 通用踩坑总结

> 跨项目反复出现的问题和教训。

### 坑 1：Excel 大文件性能问题
- **场景**: 10万+ 行达人数据用 openpyxl 读写
- **原因**: openpyxl 逐单元格操作，内存和速度都很差
- **解决**: 大文件用 CSV 中间格式过渡，最终报告用 xlsxwriter 生成
- **教训**: 选对工具比优化代码重要；数据量预估先行

### 坑 2：AI API 调用成本控制
- **场景**: Claude API 批量处理数千条数据（Bio 分类、内容分析）
- **原因**: 未做缓存和去重，每次全量调用
- **解决**: 加本地缓存层（JSON/sqlite），已处理过的数据跳过；相同 prompt 用哈希做去重
- **教训**: AI API 调用前先问"这条数据真的需要 AI 吗"，能用规则解决的不用 AI

### 坑 3：飞书 API 限流
- **场景**: 批量同步达人数据到飞书多维表格
- **原因**: 飞书 API 有 QPS 限制，连续大量写入触发限流
- **解决**: 加 `asyncio.sleep()` 延时 + 分批（每批 50 条）+ 失败重试
- **教训**: 第三方 API 集成必须考虑限流策略，不能假设"调了就能通"

## 🧠 我的设计原则

> 经过多年实践形成的个人信条。

1. **最简可用先行**：先做能用的版本，再迭代优化。不为了"以后可能"的需求加复杂度
2. **AI 是工具不是银弹**：能用规则和逻辑解决的问题不用 AI，把 AI 预算花在刀刃上
3. **数据管道要幂等**：数据处理脚本必须能重复运行不产生副作用
4. **内部工具不追求完美 UI**：功能可用 > 界面精美，飞书+本地 Web 是高效组合
